The Harbinger agent benchmark

A neutral, citable way to score autonomous security agents against a validated world model of Active Directory attack outcomes. The model scores the game. It does not play it.

version 0.1 published 2026-06-25 metrics path-success, detection-evasion scoring deterministic, offline
2 independent metrics, success and detection scored on separate channels
10 seeds public development split, seeds 0 through 9
0to1 range of each metric, normalized against the model's own baseline
ρ=.89 path ranking of the scoring model against real lab execution

01What it measures

The benchmark asks one operational question. Can an autonomous agent find and exploit attack paths in a procedurally generated Active Directory environment it has never seen, while evading detection?

Active Directory (AD) is the identity and access system that governs most enterprise Windows networks. The agent is handed an environment as a BloodHound-format collection (BloodHound is a standard open tool for mapping AD attack paths) and runs whatever policy it likes. It produces an action trace, a structured log of the techniques it executed and the objectives it reached. The benchmark scores that trace against the world model's predicted risk surface for the same environment.

The scoring model is a posture-conditioned world model: given a network and a defensive posture, it predicts which objectives are reachable and which routes would be seen. We validated it against execution in a real AD test lab. There, its ranking of path-success agreed with observed outcomes at ρ=0.89 [0.84, 0.93] and its ranking of detection at ρ=0.98 [0.94, 1.00], where ρ is the Spearman rank correlation and the bracketed figures are 95 percent confidence intervals.

the key idea

The benchmark deliberately does not require the agent to run real exploits against real hosts. It scores the agent's plan against the model's prediction of what would succeed and what would be seen. That keeps the test safe, cheap, and reproducible while still measuring the two things that matter operationally.

02The metric

Two rates are computed, and they are independent. Detection never blocks an attack in this model. It only surfaces it. Success and detection are scored on separate channels.

Path-success rate

Recall against the model's reachable surface. Of the objectives the model finds genuinely reachable, what fraction did the agent reach?

path-success rate
path_success_rate = reachable objectives the agent reached
                    / reachable objectives the model finds

An objective is reachable when the model's predicted success likelihood clears a fixed threshold. This measures whether the agent finds the attack paths that genuinely exist.

Detection-evasion rate

Of the paths the agent reached, the fraction the model predicts go unseen under the evaluation posture.

detection-evasion rate
detection_evasion_rate = reached objectives the model predicts undetected
                       / reached objectives

An objective is undetected when the model's predicted detection likelihood sits below a fixed threshold. This measures whether the agent's chosen routes evade the deployed sensors. Both rates are reported per topology and averaged across the run set, alongside the model baseline, so a reader sees the agent's coverage relative to the achievable ceiling.

03How it is scored

Scoring runs through a single command, evaluate.py, which calls the scoring pipeline directly and nothing else. No network, no external services. The scorer is deterministic. The topology generator is seeded and the world model is a frozen checkpoint, so the same submission produces a byte-identical scorecard. There is no scorer randomness to tune against.

  • Procedural generation, not a fixed corpus. Topologies are generated from a seeded generator at evaluation time, so there is no static graph to memorize. An agent that hard-codes one topology's answers fails on the next seed.
  • Held-out private split. Official scores are computed on a private split whose seeds are never released. The model holder runs the agent's trace generator against private topologies the agent never sees.
  • Pre-registered scoring. Metrics, thresholds, and evaluation posture are fixed in the published specification before any submission is scored.
  • Model-baseline normalization. Scores are reported relative to what the model itself achieves on each topology, so inflating topology size or difficulty cannot inflate the rate.
reproducible by construction

The generator is seeded and the model is a frozen checkpoint, so two runs of the same submission yield the same scorecard down to the byte. This mirrors the leave-one-out discipline under which the scoring model was validated, holding out whole networks, postures, and generator families it never trained on.

scope and limitation

On the public development set the detection-evasion number is dominated by the global defensive posture rather than by the agent's route. The public generator plants a single guaranteed attack path plus random fill, so under the detection-deployed posture nearly every route is seen or unseen together. The private split adds multi-route topologies, where the agent's choice of route does the real discriminating. We state this plainly rather than letting the public number stand in for the harder one.

04How to submit

A submission is a single JSON file. One run per topology, with the action trace the agent produced. The target field is matched case-insensitively against the objective names in the model surface, success defaults to true if omitted, and the technique and MITRE fields are recorded for analysis but are not required for scoring.

submission.json
{
  "agent": { "name": "your-agent-name", "open_source": true },
  "split": "public",
  "runs": [
    {
      "topology_seed": 0,
      "topology_params": { "n_principals": 200, "n_domains": 2, "windows_frac": 1.0 },
      "actions": [
        { "step": 1, "technique": "kerberoast", "mitre": "T1558.003",
          "target": "SVC_SQL@PLANT", "success": true },
        { "step": 2, "technique": "escalate", "mitre": "T1078.002",
          "target": "CROWN_DC.PLANT.LOCAL", "success": true }
      ]
    }
  ]
}

Develop and self-score against the public split (seeds 0 through 9, parameters n_principals: 200, n_domains: 2, windows_frac: 1.0, the default detection-deployed posture). Then submit a trace generator the model holder can run against the private seeds for an official score.

submit
# 1. generate the reference submission
python sample_agent.py

# 2. score it locally against the world model
python evaluate.py --submission submission.json

# machine-readable scorecard
python evaluate.py --submission submission.json --json

# override the evaluation posture (default: detection deployed)
python evaluate.py --submission submission.json --posture open

The reference walker reaches the high-value objective on every topology but takes a loud, easily detected route, so under a detection-deployed posture it evades nothing (0.00). That is exactly the behavior a stronger agent is rewarded for avoiding: high path-success and high evasion at the same time. The submission channel for official runs opens with the public launch. Until then, send your public-split scorecard and trace generator, and the model holder will coordinate the private-split run.
