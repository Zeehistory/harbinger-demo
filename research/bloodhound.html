01Why these queries

BloodHound shows you what paths exist. It maps reachability across an Active Directory forest, the directory service that governs identity and access on Windows networks, and lets you walk the graph by hand. What it does not do is rank. Every reachable path looks equal until a person decides which one to chase. Harbinger adds a ranking layer on top of the same collection, scoring paths by which ones are likely to succeed under the deployed security posture and which ones are likely to evade detection.

These seven queries sit in between. They surface the exact objects and edges that ranking reasons over, so you can see them in BloodHound directly, with or without running any further analysis. They are offered as a contribution to the ecosystem. There is no further analysis without a SharpHound collection first. SharpHound is the collector that produces the BloodHound graph, and these queries make that collection more legible.

All seven import under a single Harbinger category in the BloodHound query list. 4 return graph paths you can expand and pivot through. The other 3 return tables of accounts with the attributes that decide priority, such as Kerberos encryption type and the date a password was last set. Kerberos is the authentication protocol Active Directory uses.

the complement

BloodHound shows what is reachable. These queries help surface what is likely to matter. A ranking layer can then sort the reachable paths by whether they are likely to succeed under the deployed posture and whether the logs are likely to record them, and point to the change that reduces the most risk.

02The seven queries

Each query targets a class of misconfiguration that often shapes an engagement: kerberoastable service accounts, dangerous write access onto high-value objects, unconstrained delegation, live sessions that bridge a foothold to the core, and the cross-domain trust edges along which a single compromise can travel. Kerberoasting is a technique that requests service tickets for accounts with a registered service principal name and cracks them offline.

Three of these are worth reading in full, because the filter is where the priority lives. The first ranks roastable accounts by their encryption type, putting weaker crypto at the top of the list. The next finds the session bridges that connect a workstation foothold to a domain takeover. The last maps the trust edges that determine how far a single compromise can travel.

kerberoastable SPN accounts, with encryption type 
MATCH (u:User)
WHERE u.hasspn = true AND u.enabled = true
RETURN u.name AS account,
       u.serviceprincipalnames AS spns,
       u.supportedencryptiontypes AS encryption_type,
       u.pwdlastset AS pwd_last_set
ORDER BY u.supportedencryptiontypes
session-bearing paths to Domain Admins 
MATCH p = (c:Computer)-[:HasSession]->(u:User)
          -[:MemberOf|AddMember|GenericAll|GenericWrite|WriteDacl|Owns*1..]->(g:Group)
WHERE g.objectid ENDS WITH '-512'
RETURN p
cross-domain trust edges, forest map 
MATCH p = (a:Domain)-[r:TrustedBy]->(b:Domain)
RETURN p

03How to import

The query set ships as bloodhound_queries.json. It surfaces the objects and edges the queries describe and is useful on its own, whether or not you run any further analysis. If you already keep custom queries, merge the queries array rather than overwriting. The new ones appear under the Harbinger category.

import 
# Legacy BloodHound (Electron GUI): copy the file, then restart the app.
# Windows
%USERPROFILE%\AppData\Roaming\bloodhound\customqueries.json
# macOS
~/Library/Application Support/bloodhound/customqueries.json
# Linux
~/.config/bloodhound/customqueries.json

# BloodHound CE: paste any query's Cypher into the search bar,
# or add it via Settings -> Custom Queries.

To produce a complete export, run SharpHound with session and local-admin collection enabled. Those edges are the foothold-to-core bridges. Without them, the analysis reports a collection gap rather than a false claim of no path.

collect, then analyze 
SharpHound.exe -c All,Session,LoggedOn,LocalAdmin
harbinger analyze --export bloodhound.json
on scope

The ranking layer has been validated against real attack execution on a lab forest with native Windows event-log ground truth. It has not yet been validated against production endpoint-detection systems or real enterprise networks. The queries above are plain Cypher and carry no such caveat. They run anywhere BloodHound does.

BloodHound answers whether a path can be walked. The ranking layer estimates whether it is likely to succeed under the deployed posture, whether the logs are likely to record it, and which change reduces the most risk. It is additive to the BloodHound ecosystem, not a replacement. It consumes the export and adds a layer. It does not change how you collect or visualize.
