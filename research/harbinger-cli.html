free local tool

harbinger-cli

A command-line tool that reads a BloodHound or SharpHound export, the JSON inventory those tools produce of an Active Directory environment, and ranks which Active Directory attack paths are likely to succeed, which are likely to evade detection, and what to fix first. It runs entirely on your machine.

license free, open source python 3.10 or newer privacy no data leaves your machine version 0.1.0
runs offline no telemetry no api calls open source no new dependencies confidence intervals on every number

What it does

harbinger runs a predictive model of Active Directory (AD) attack outcomes against a collection export. The model is conditioned on the defensive posture of the environment, so its predictions reflect what defenses are in place. The tool runs four stages in order: it reads the export, scores how complete the collection is, ranks the attack paths by risk, and writes a report. For a given export it tells you the collection completeness tier first, then the top attack paths ranked by reachable-and-undetected risk, then the highest-impact fixes.

Every figure carries a confidence interval (CI), the range the true value is expected to fall within. Every report ends with three open limitations printed in full, so the limits of the model travel with its output rather than getting lost.

4 commands: analyze, score, check, download-model
3 validated fix classes: rotate credentials, lock delegation, segment
0 deps no new external dependencies beyond the repository
50 nodes generated graph the self-test runs on, no file needed

step 01Install

Clone the repository, install the editable package, and the harbinger command becomes available. No new external dependencies are declared; the tool reuses the libraries the repository already pins.

install the command 
git clone https://github.com/harbinger-labs/world-model-for-cybersecurity
cd world-model-for-cybersecurity/harbinger-cli
pip install -e .                 # installs the `harbinger` command

Then fetch the model, which is large and downloaded on demand rather than bundled with the code, and verify the install with the self-test.

fetch the model, then verify 
harbinger download-model
harbinger check

step 02analyze

Point analyze at a BloodHound or SharpHound export. It prints the completeness tier first (full, partial, or insufficient, with specific re-collection guidance when not full), then the top paths ranked by reachable-and-undetected risk, then the top three fixes by change in risk. Optional flags are --output to write the report to a file, --top for how many paths to show, --posture for a named defensive posture, and --threat with choices worst_case or typical.

harbinger analyze 
harbinger analyze --export bloodhound.json
harbinger analyze --export bloodhound.json --output report.md --top 10

step 03score

score answers a what-if question: what a defensive change does to the attack surface. It shows the before and after paths and the predicted change in risk, with a deliberately wider interval to reflect the added uncertainty of a what-if estimate. The change is labelled validated for the three classes that have been tested against lab outcomes (rotate credentials, lock delegation, segment the network) and out-of-validation for anything else. A name after the colon, such as :jdoe, is a scoping label. The model applies the change at the posture level, so naming a single account is a label, not a separately validated setting.

harbinger score 
harbinger score --export bloodhound.json --intervention "rotate_creds:jdoe"
harbinger score --export bloodhound.json --intervention "lock_deleg"
harbinger score --export bloodhound.json --intervention "segment"

# Counterfactual. rotate_creds:jdoe
**Fix class: rotate credentials. validated**

## Risk before / after
- before: 0.992
- after:  0.214
- change in risk = -0.778  (90% ci [-0.91, -0.55])

step 04check

check runs the full analysis on a generated 50-node graph, so no file is needed. It exercises every stage in order: generate, read, score completeness, rank risk, report. Use it to confirm a fresh install is healthy.

download-model

The model is large and downloaded on demand rather than shipped with the code. download-model places it locally, skipping anything already present. Pass --force to re-download. The download location can be pointed at a fork or private mirror with the HARBINGER_MODEL_URL environment variable.

harbinger download-model 
harbinger download-model
harbinger download-model --force      # re-download even if present

  [fetch]   model file  <-  release download
  [ok]      model file
  [present] model file (already local)

model ready.

Validation, in numbers

The model was tested against a real run on a Game of Active Directory (GOAD) lab, a public Active Directory test environment. We measured the rank correlation, written ρ, between the model's predicted ranking and the observed outcomes. A value of 1.0 means the orderings match exactly. The 90% confidence intervals are carried below.

ρ=0.89090% ci [0.843, 0.934]path-success ranking
ρ=0.97590% ci [0.943, 1.000]detection ranking
path-success rank correlation 0.890
detection rank correlation 0.975

Path-success correlation is ρ=0.890 with a 90% interval of [0.843, 0.934]. Detection correlation is ρ=0.975 with [0.943, 1.000]. These results hold on lab environments. The limitations below say where they do not.

the core promise

harbinger is free and runs entirely on your machine. No data leaves the host, no network calls during analysis, no telemetry. The model is a local file and the analysis is a local computation. Run it first against an environment you already understand, your own GOAD lab or a test export, and verify the output before pointing it at anything real.

caveat: model download required

The model is not bundled with the code. analyze and score will stop with a clear message until you run harbinger download-model. If the default download location is unreachable, set HARBINGER_MODEL_URL to a reachable mirror or place the model file locally yourself.

The three open limitations

Every report ends with these three limitations, carried in full. They are the honest edge of what the model has been shown to do.

  1. Real enterprise networks are not validated. Results are from lab environments only.
  2. Production endpoint detection and response (EDR) tools, the agents that watch hosts for malicious behavior, are not validated. Lab detection partially closes this gap, but lab detection is not production-grade EDR.
  3. Behavior over time and real network change are not validated. Confirming the model holds on real, time-stamped exports remains the decisive test.
